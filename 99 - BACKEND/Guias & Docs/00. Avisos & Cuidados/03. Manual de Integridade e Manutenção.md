

Este documento define as regras operacionais para garantir a estabilidade do sistema. O S.O.T.A. opera como uma aplicação de software sobre o Obsidian, dependendo de caminhos de arquivos fixos (hardcoded) e padrões de nomenclatura específicos para vincular dados, gerar gráficos e calcular métricas.

A violação destas regras resultará em perda de vínculo entre dashboards e dados históricos.

---

## 1. Estrutura Crítica (Não Modificar)

Os diretórios abaixo possuem caminhos escritos diretamente no código-fonte dos scripts e do plugin. Alterar o nome ou mover estas pastas quebrará o sistema imediatamente.

### A Pasta Backend
*   **Caminho:** `99 - BACKEND`
*   **Função:** Contém o banco de dados de logs (`Logs_Metricas`), os scripts de automação, templates e recursos visuais.
*   **Regra:** Jamais renomeie, mova ou apague esta pasta. Ela é a raiz de todas as operações de leitura e escrita do sistema.

### O Banco de Dados de Exercícios
*   **Caminho:** `04 - Corpo & Movimento/01 - Exercícios Físicos`
*   **Subpastas Críticas:** `01. Manual/Exercícios` e `01. Manual/Grupos Musculares`.
*   **Regra:** Os scripts de agendamento de treino e renderização de gráficos buscam exercícios especificamente nestes caminhos. Não altere a hierarquia interna destas pastas.

### Arquivos de Log e Métricas
*   **Arquivos:** `raw_logs.md` e `_metrics.md`.
*   **Regra:** Estes arquivos são gerados automaticamente pelo sistema. A edição manual é desencorajada, pois dependem de sintaxe estrita para serem lidos pelos algoritmos de ETL (Extract, Transform, Load).

## 2. Convenção de Nomenclatura para Mídias (Numeração)

O sistema utiliza uma lógica de pastas para organizar Cursos, Séries e Documentários. Para que os scripts consigam identificar a ordem correta de Módulos, Temporadas ou Capítulos, é obrigatório seguir o padrão numérico.

### A Regra do Prefixo Numérico
Ao criar ou renomear pastas dentro de Mídias (especialmente Cursos e Séries), mantenha o formato:

`XX - Nome Descritivo`

**Exemplos Corretos:**
*   `01 - Introdução`
*   `02 - Fundamentos Avançados`
*   `10 - Conclusão`

**Exemplos Incorretos (O sistema falhará ao ordenar):**
*   `Introdução` (Sem número)
*   `1. Introdução` (Formato de ponto pode variar)
*   `Módulo 1` (Texto antes do número)

**Por que isso é necessário?**
Os scripts `renderGuiaModulos.js` e os dashboards de progresso utilizam expressões regulares (Regex) para extrair os dois primeiros dígitos do nome da pasta. Isso define a ordem de exibição nas tabelas e garante que o "Módulo 02" venha depois do "Módulo 01", e não em ordem alfabética aleatória.

## 3. Gerenciamento de Projetos e Identidade (UID)

O sistema utiliza um identificador único (`sota_uid`) no cabeçalho (Frontmatter) de cada nota para manter os vínculos, independentemente do nome do arquivo.

### Renomeação de Projetos e Mídias
*   **Procedimento:** Não utilize a função "Renomear" do explorador de arquivos do Obsidian para pastas de Projetos ou Estudos ativos.
*   **Risco:** Ao renomear a pasta de um projeto manualmente, o Obsidian não move a pasta de logs correspondente dentro do `99 - BACKEND`. O vínculo é quebrado e o histórico de dados desaparece do dashboard.
*   **Solução:** Utilize os botões de script "Renomear" disponíveis no painel de controle do projeto. Estes scripts executam a renomeação segura, movendo tanto a pasta do projeto quanto a pasta de logs oculta.

### O Campo `sota_uid`
*   **Local:** No topo do arquivo, entre os `---`.
*   **Regra:** Nunca apague ou altere manualmente o código alfanumérico deste campo. Ele é a chave primária que conecta a nota aos seus logs, tarefas e gráficos.

---

## 4. Edição de Logs Diários

A nota diária (`01 - Registros/01. Daily`) atua como a interface de entrada de dados.

*   **Sintaxe de Log:** O sistema processa linhas que contêm chaves específicas, como `(sessao_fim::WORK)` ou `(log_time::HH:mm:ss)`.
*   **Edição Permitida:** Você pode corrigir erros de digitação no texto descritivo da tarefa.
*   **Edição Proibida:** Não altere os valores dentro dos parênteses de metadados gerados automaticamente. Alterar um timestamp ou remover uma tag de identificação fará com que aquela sessão seja ignorada nos cálculos de performance do dia.

---

## Resumo Operacional

1.  **Imutabilidade do Backend:** A pasta `99 - BACKEND` é estática.
2.  **Numeração de Mídias:** Sempre use `01 - Nome` para pastas de módulos/temporadas.
3.  **Renomeação via Script:** Use os botões do sistema para renomear estruturas complexas.
4.  **Preservação de UID:** O `sota_uid` no Frontmatter é intocável.
